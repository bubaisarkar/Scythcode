# Firestore Setup for Ticket System

## Why We Switched from SQLite to Firestore

SQLite doesn't work on Vercel because:
- Vercel uses serverless functions (no persistent file system)
- Each request runs in isolation
- No way to maintain a database file

Firestore is perfect for Vercel because:
- Cloud-based (no local files needed)
- Real-time updates
- Serverless-friendly
- Already using Firebase for auth

## Setup Steps

### 1. Enable Firestore in Firebase Console

1. Go to [Firebase Console](https://console.firebase.google.com/)
2. Select your project: **scythcode**
3. Click "Firestore Database" in the left sidebar
4. Click "Create database"
5. Choose **"Start in production mode"**
6. Select a location (choose closest to your users, e.g., `us-east1`)
7. Click "Enable"

### 2. Set Up Security Rules

In the Firestore console, go to the "Rules" tab and replace with:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Tickets collection
    match /tickets/{ticketId} {
      // Users can read their own tickets
      allow read: if request.auth != null && 
                     resource.data.user_id == request.auth.uid;
      // Users can create tickets
      allow create: if request.auth != null &&
                       request.resource.data.user_id == request.auth.uid;
      // Users can update their own tickets (limited fields)
      allow update: if request.auth != null && 
                       resource.data.user_id == request.auth.uid;
    }
    
    // Ticket messages collection
    match /ticket_messages/{messageId} {
      // Users can read messages for their tickets
      allow read: if request.auth != null;
      // Users can create messages
      allow create: if request.auth != null;
    }
  }
}
```

**Important:** These rules allow users to only see their own tickets. Admin access is handled through the API routes with password authentication.

### 3. Create Indexes (Optional but Recommended)

Firestore will prompt you to create indexes when needed. The first time you filter tickets by status or user, Firebase will show an error with a link to create the index automatically.

Common indexes needed:
- Collection: `tickets`, Fields: `user_id` (Ascending), `created_at` (Descending)
- Collection: `tickets`, Fields: `status` (Ascending), `created_at` (Descending)
- Collection: `ticket_messages`, Fields: `ticket_id` (Ascending), `created_at` (Ascending)

### 4. Test the Ticket System

1. Go to your deployed site on Vercel
2. Log in with your Firebase account
3. Go to "My Tickets" or `/tickets/create`
4. Create a test ticket
5. Verify it appears in Firebase Console > Firestore Database

## Collections Structure

### tickets
```javascript
{
  id: string,                    // Auto-generated by Firestore
  ticket_number: string,         // SCT-12345678-123
  user_id: string,               // Firebase UID
  user_email: string,
  user_name: string,
  subject: string,
  category: string,
  priority: string,
  status: string,                // open, in-progress, resolved, closed
  created_at: timestamp,
  updated_at: timestamp,
  closed_at: timestamp | null
}
```

### ticket_messages
```javascript
{
  id: string,                    // Auto-generated by Firestore
  ticket_id: string,             // References ticket document ID
  sender_type: string,           // user, admin, system
  sender_name: string,
  sender_email: string | null,
  message: string,
  created_at: timestamp
}
```

## Troubleshooting

### "Missing or insufficient permissions" error
- Check Firestore security rules are set up correctly
- Make sure user is authenticated before accessing tickets

### Index required error
- Click the link in the error message to create the index automatically
- Or create manually in Firestore Console > Indexes tab

### Tickets not showing up
- Check Firebase Console > Firestore Database to verify data is being written
- Check browser console for errors
- Verify user is logged in with Firebase Auth

## Benefits of Firestore

✅ Works perfectly on Vercel (serverless)
✅ Real-time updates (no need for polling)
✅ Automatic scaling
✅ No database maintenance
✅ Built-in security rules
✅ Works with existing Firebase Auth
